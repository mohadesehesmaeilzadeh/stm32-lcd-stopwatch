# STM32 StopWatch Project Review

## Scope Reviewed

Reviewed artifacts:

- STM32CubeIDE project from `StopWatch.zip`
- Main firmware source under `Core/Src` and headers under `Core/Inc`
- Linker scripts and generated Debug build files
- Proteus project `StopWatch.pdsprj`
- PDF report `Negar&Mohadeseh,HW4.pdf`

The active project is an STM32F401VETx stopwatch implemented mostly with direct register access. It uses TIM2 for stopwatch timing, SysTick for helper timing, EXTI4 for the Start/Pause button, an LCD1602 in 4-bit mode, a speed-change button, an LED, and a buzzer.

## High-Level Architecture

```text
                       +---------------------------+
                       | STM32F401VETx MCU         |
                       | ARM Cortex-M4 core        |
                       +-------------+-------------+
                                     |
             +-----------------------+------------------------+
             |                       |                        |
       +-----v-----+           +-----v------+           +-----v------+
       | Clock HSI |           | SysTick    |           | TIM2       |
       | 16 MHz    |           | 1 ms tick  |           | 1 ms IRQ   |
       +-----+-----+           +-----+------+           +-----+------+
             |                       |                        |
             |                       |                        |
       +-----v-----------------------v------------------------v------+
       | Main loop and interrupt handlers                            |
       | - Start/Pause state                                         |
       | - LED speed state                                           |
       | - Buzzer rule                                               |
       | - LCD refresh every 50 ms                                   |
       +-----+------------+------------+------------+----------------+
             |            |            |            |
       +-----v--+   +-----v--+   +-----v--+   +-----v------+
       | LCD    |   | Buttons |   | LED    |   | Buzzer     |
       | PB pins|   | PE4/PE5 |   | PC13   |   | PB6        |
       +--------+   +--------+   +--------+   +------------+
```

Runtime flow:

```text
Reset
  -> startup_stm32f401vetx.s initializes stack/data/BSS
  -> main()
      -> Clock_Init(): selects HSI, assumes 16 MHz
      -> SysTick_Init(): creates sys_ms helper counter
      -> GPIO_Init(): configures LCD, LED, buzzer, buttons
      -> TIM2_Init(): creates 1 ms stopwatch interrupt
      -> EXTI4_Init(): enables Start/Pause button interrupt
      -> LCD_Init()
      -> forever:
           read stopwatch_ms
           poll PE5 speed button
           toggle LED when led_delay_ms expires
           beep on new even stopwatch second
           refresh LCD every 50 ms

Interrupts:
  SysTick_Handler -> increments sys_ms
  TIM2_IRQHandler -> increments stopwatch_ms only when running
  EXTI4_IRQHandler -> debounced toggle of stopwatch_running
```

## Microprocessor / Microcontroller Interaction

This is not a custom microprocessor with a custom ALU/register-file/control-unit design. It is firmware running on an ARM Cortex-M4 microcontroller. The relevant internal parts interact like this:

- ALU: executes arithmetic and logic instructions, such as millisecond increments, modulo/division for seconds, comparisons, and GPIO bit manipulation.
- Core registers: hold temporary values during execution, function parameters, return addresses, and interrupt context.
- Special registers: NVIC, SysTick, RCC, GPIO, TIM2, EXTI, and SYSCFG registers control the MCU peripherals.
- Flash memory: stores firmware code, vector table, constants, and interrupt handlers.
- SRAM: stores global variables such as `sys_ms`, `stopwatch_ms`, `stopwatch_running`, `led_delay_ms`, and LCD buffers.
- Clock system: uses internal HSI at 16 MHz in the hand-written firmware. SysTick and TIM2 timing calculations depend on this value.
- NVIC/control unit: dispatches interrupt handlers for SysTick, TIM2, and EXTI4.
- GPIO peripheral registers: drive the LCD, LED, and buzzer, and read push-button inputs.

Instruction set: the MCU executes the ARM Thumb-2 instruction set generated by GCC. The project itself does not define a custom instruction set. The application-level commands are Start/Pause, Change LED Speed, Timer Tick, LCD Update, and Buzzer Trigger.

## Main Firmware Behavior

Pin mapping from the active `main.c`:

| Function | MCU pin |
|---|---|
| LCD RS | PB0 |
| LCD E | PB1 |
| LCD D4 | PB2 |
| LCD D5 | PB10 |
| LCD D6 | PB13 |
| LCD D7 | PB12 |
| Buzzer | PB6 |
| LED | PC13 |
| Start/Pause button | PE4 / EXTI4 |
| Speed button | PE5 |

The stopwatch time is stored in `stopwatch_ms`. TIM2 increments it every 1 ms, but only when `stopwatch_running` is true. The Start/Pause button toggles `stopwatch_running` through EXTI4 with a 200 ms debounce. The second button cycles the LED blink period through 3000, 1500, 750, and 375 ms. The buzzer gives a short 100 ms pulse when the running stopwatch reaches a new even-numbered second.

## Strengths

- Clear direct-register implementation: good for a microprocessor/microcontroller course because the code exposes RCC, GPIO, TIM2, EXTI, SysTick, and NVIC usage.
- TIM2 handles stopwatch timing, so stopwatch time is not based on blocking delay loops.
- The main loop keeps higher-level processing outside the timer ISR, especially the even-second buzzer decision.
- Button handling includes software debounce.
- LCD is abstracted into small functions, making the main loop readable.
- The PDF documents the circuit, pin mapping, timer calculation, and major debugging steps.
- Existing build output contains `StopWatch.hex`, so the simulation can run even if a clean build is not performed immediately.

## Weaknesses, Risks, and Potential Bugs

1. Build portability issue: `Debug/makefile` references an absolute linker path under `C:\Users\asus\Downloads\StopWatch\STM32F401VETX_FLASH.ld`. A clone on another machine will not build from that makefile without editing or regenerating the project.

2. Source tree appears incomplete for a clean standalone build: the root source tree does not include a full `Drivers/` folder, while the generated makefiles include CMSIS/HAL include paths. A GitHub version should either include the required generated `Drivers/` tree or explain that STM32CubeIDE must regenerate it from `StopWatch.ioc`.

3. Generated CubeMX configuration and active code disagree. `StopWatch.ioc` still describes PLL/84 MHz style settings, `tim.c` uses `Prescaler = 83`, and `gpio.c` sets PE4 as rising-edge EXTI. The active firmware uses HSI 16 MHz, TIM2 `PSC = 15`, `ARR = 999`, and falling-edge EXTI. This can confuse future maintainers.

4. The project mixes two styles: CubeMX/HAL-generated modules are compiled, but the application intentionally avoids HAL and implements handlers/init in `main.c`. This works, but it increases the chance of duplicate interrupt handlers or conflicting settings after CubeMX regeneration.

5. LCD writes are blocking and relatively slow. `Update_LCD()` rewrites both 16-character lines every 50 ms, and each character uses delay-based LCD timing. This does not break the TIM2 interrupt, but it keeps the foreground loop busy and can make button polling less responsive.

6. `sprintf()` pulls in formatted I/O code from newlib. The map file confirms libc formatting code is linked. For small embedded firmware, manual formatting or `snprintf()` is usually better.

7. `sprintf()` is not bounded. The current format strings fit the 17-byte buffers, but `snprintf()` would protect the code if the display text changes later.

8. The comments in `main.c` show mojibake in the terminal, which suggests an encoding mismatch. Save the file consistently as UTF-8 to keep Persian comments readable across tools.

9. The direct buzzer drive is fine in Proteus, but real hardware should usually use a transistor driver and protection as needed, depending on buzzer type and current.

10. PC13 LED polarity depends on the board/circuit. In this Proteus design it is treated as a normal active-high output through an LED/resistor path. Many STM32 boards wire PC13 LEDs active-low, so this should be documented for real hardware.

11. No reset/lap/clear feature is implemented. That is acceptable if not required, but a stopwatch usually benefits from a reset button or long-press reset behavior.

12. The PDF starts with a separate seven-segment MM:SS project before the current LCD stopwatch project. For a public repository, the README should focus only on the final stopwatch or clearly separate the two exercises.

## Optimization Suggestions

- Replace `sprintf()` with `snprintf()` or a small manual decimal formatter.
- Only update LCD fields that changed instead of rewriting both full rows every 50 ms.
- Move LCD, button, LED, buzzer, clock, and timer code into separate `.c/.h` modules for readability.
- Use symbolic constants for debounce times, LCD width, buzzer pulse time, and LED speed values.
- Align CubeMX `.ioc` settings with the final direct-register configuration or document that the `.ioc` is only a pin-planning aid.
- Add a `README.md`, `.gitignore`, and a short `docs/wiring.md`.
- Do not commit `Debug/` object files, `.elf`, `.map`, `.list`, `.o`, `.d`, `.su`, and `.cyclo` files.
- For GitHub reproducibility, include either the generated `Drivers/` folder or detailed regeneration steps from STM32CubeIDE.

## Technical Summary

This project is a register-level STM32F401VETx stopwatch. TIM2 generates a 1 ms interrupt and increments a millisecond counter while the stopwatch is running. EXTI4 handles a debounced Start/Pause button on PE4. SysTick provides a separate 1 ms helper timebase for delays, LCD timing, button debounce, LED scheduling, and buzzer pulse timing. The LCD is driven in 4-bit mode using PB0, PB1, PB2, PB10, PB13, and PB12. A second button on PE5 cycles the LED blink period, and a buzzer on PB6 pulses for 100 ms on new even stopwatch seconds.

## Non-Technical Summary

The project is an electronic stopwatch. One button starts and pauses the time. The current time is shown on a small LCD screen. A light blinks at a speed that can be changed with a second button. A buzzer makes a short sound when the stopwatch reaches an even-numbered second. The circuit was tested in Proteus simulation.
